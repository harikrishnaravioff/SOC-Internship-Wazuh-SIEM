# Week 2 — Log Monitoring & Threat Hunting

## Executive Summary

The objective of Week 2 was to move beyond basic agent connectivity and begin analyzing the security data generated by the Kali Linux endpoint. After successfully deploying the Wazuh Manager and Agent in Week 1, the focus shifted to monitoring system logs, investigating authentication-related events, and understanding how Wazuh organizes security information.

Since the monitored endpoint was running Kali Linux, authentication monitoring relied on Linux system logs, PAM records, and SSH-related events. By analyzing the collected logs, it was possible to identify failed login attempts and other security-related activities, providing practical experience with log analysis and threat hunting.

---

## Methodology

The monitoring process began by accessing the Threat Hunting module within the Wazuh Dashboard and filtering events generated by the Kali Linux agent. Searches were performed using the agent name to ensure only data from the monitored endpoint was displayed.

Several queries were used to investigate authentication activity:
- Authentication failure searches to identify unsuccessful login attempts
- Broader authentication-related searches to capture additional login events
- SSH-related filters since SSH is the primary access method on Linux systems

A dedicated group was also created for the Kali Linux agent instead of relying solely on the default group, allowing easier management as additional agents are added in the future. The Events section was reviewed to examine raw logs collected by Wazuh, providing visibility into authentication events, rootcheck findings, security configuration assessments, and other endpoint activity.

---

## Technical Findings

The analysis identified **10 authentication-related failure events** during the monitoring period. None of these events reached a critical severity level, indicating that although failed login attempts occurred, no major security incident was detected.

All authentication events observed were failed login attempts with no successful authentications recorded. The Threat Hunting dashboard mapped these events to MITRE ATT&CK techniques including:

| MITRE Technique | Category |
|----------------|----------|
| Password Guessing | Largest category |
| SSH Activity | Secondary finding |
| Brute Force | Supporting evidence |

Additional searches focused on SSH-related activity and produced results consistent with the authentication failure findings. Beyond authentication monitoring, the event stream included rootcheck alerts, Security Configuration Assessment findings, CIS benchmark results, PAM session activity, and sudo-related events.

## Recommendations

- Monitor authentication failures to determine their source and frequency to distinguish between user errors and brute-force activity
- Implement custom alert rules and active response actions for repeated authentication failures
- Consider automated actions such as temporarily blocking suspicious IP addresses after multiple failed login attempts
- Review and address Security Configuration Assessment findings to improve system hardening
- Create dedicated dashboards and saved searches for specific agent groups as more agents are added
